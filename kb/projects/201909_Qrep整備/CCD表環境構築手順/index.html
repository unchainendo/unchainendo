CCD表の環境を整備する
2020/03/01

作業前提

背景

ホストからDBサーバーへのQレプリケーション環境が完成した後に以下の追加要件が発生したのでCCD環境の構築手順を整理した。 最終的には構築せずに終わってしまったが、手順は検証環境での稼働確認の上で作成したものなので今後のために残しておく。

  • ターゲットDBは本来参照しかしない想定だが、あるテーブルでターゲット側のレコードが「NOT FOUND」となる事象が発生した。(EXCEPTIONS表にエラーが記録された)
  • ターゲット表のレコードを何か(誰か)が削除した可能性があるため、ターゲット表の更新履歴を残せる仕組みがほしい。
  • Db2の監査機能(db2audit)を使用するとデータベース単位でログが収集されてしまい、かつQレプリケーションによる正常な更新処理も全て記録されるため ログ保管領域の管理やログの消込の実装が必要になり非常に手間がかかる。
  • そこでターゲットDBに対してQキャプチャーを仕掛け、特定のテーブルに対するQレプリケーション以外の変更をCCD表として吐き出す仕組みを構築したいということになった。

実装方針

CCD環境を整備するにあたって、既に実装済みのQレプリケーション環境への影響等を考慮して以下の実装方針とすることとした。

  • DBサーバー上のQrep用ユーザーはCCD環境用に新しいものを作成し、そのユーザーに紐づくスキーマにQアプライとQキャプチャーの新しい制御表を作成する。
  • IDRサーバー上のQアプライ実行ユーザー(db2qrep)はCCD環境用のQキャプチャー/アプライ実行ユーザーとしても使用する。ただしログやパスワードファイル、IPCファイル等の出力先(主にcapture_path, apply_path)はディレクトリを新規作成して分けて管理する。(一緒にすると管理が煩雑になるため。)
  • MQについてはCCD環境用にキュー・マネージャーを分けて作成する必要はないためCCD環境のMQオブジェクトの作成先としても利用(共用)する。 キュー・マネージャーを共用することでデータパス(/var/mqm/qmgrs)やログパス(/var/mqm/log)も共用することとなり、本来はファイルシステムサイズの再設計が必要となるところだが、今回は対象が1テーブルのみで、かつCCD表に格納するのはQレプリケーションによる更新「以外」の変更履歴のみとなるため、もともと余裕を持って設計しているデータパスとログパスはそのまま共用することとする。

環境区分

  • IDRサーバー Qキャプチャー、Qアプライを稼働させる環境
  • DBサーバー (本来のQレプリケーション環境における)ターゲットDBが存在する環境

ソフトウェアバージョン

  • IDRサーバー
    • Db2 Standart Edition V11.5
    • IDR for Availability V1.2 (ARCH_LEVEL=1140)
    • MQ V9.1
  • DBサーバー
    • Db2 V11.5

1. 事前準備

CCD環境用Qrep制御表スキーマ・ユーザー作成

DBサーバー上でCCD環境用のQキャプチャーとQアプライの制御表用のスキーマと紐づくユーザーを作成する。 本来のQレプリケーション環境でアプライ制御表のためのスキーマ(ユーザー)はqrep_targetとしていたためCCD環境用のスキーマ(ユーザー)はqrep_ccdとする。1つのスキーマでQキャプチャーとQアプライの制御表を作成することはできるが、複数のQアプライの制御表を作成することはできないため今回スキーマ(ユーザー)は必ず作らないといけない。 また、DBサーバー上のテーブルにアクセスできるようにするためqrep_targetユーザーと同様にターゲットDBのDBADM権限を付与しておくこと。

2. Db2設定

Qアプライ実行ユーザー環境設定

IDRサーバー上のQアプライ実行ユーザー(db2qrep)はCCD環境としても共用するが、ログやIPCファイル、パスワードファイルの格納場所は分けておきたいので、CCD環境専用のディレクトリを作成してそこをcapture_path、apply_pathとする。 以降の作業は基本的にQアプライ実行ユーザー(db2qrep)で実施する。

# 定義関連スクリプトの配置先ディレクトリを作成
cd /home/db2qrep/QREP/define

# キャプチャー制御表作成スクリプトを作成
cat > ./create_ccd_capture_ctltable.txt << EOF
ASNCLP SESSION SET TO Q REPLICATION;
SET PWDFILE "<パスワードファイル(フルパス)>";
SET SERVER CAPTURE TO DB <ターゲットDB名>;

SET QMANAGER <キュー・マネージャー名> FOR CAPTURE SCHEMA;
SET CAPTURE SCHEMA SOURCE <制御表スキーマ名>;
SET OUTPUT CAPTURE SCRIPT "capture_ccd_ctltable.sql";
SET OUTPUT TARGET SCRIPT "apply_ccd_ctltable.sql";

#SET RUN SCRIPT LATER;
SET RUN SCRIPT NOW STOP ON SQL ERROR ON;

CREATE CONTROL TABLES FOR CAPTURE SERVER USING RESTARTQ <再始動キュー名> ADMINQ <管理キュー名> MONITOR LIMIT 93600 MONITOR INTERVAL 60000 IN UW TBSPACE <制御表用の表スペース>;
EOF

# キャプチャー制御表作成
asnclp -f ./create_ccd_capture_ctltable.txt

# アプライ制御表作成スクリプトを作成
cat > ./create_ccd_apply_ctltable.txt << EOF
ASNCLP SESSION SET TO Q REPLICATION;
SET PWDFILE "<パスワードファイル(フルパス)>";
SET SERVER TARGET TO DB <ターゲットDB名>;

SET QMANAGER <キュー・マネージャー名> FOR APPLY SCHEMA;
SET APPLY SCHEMA <制御表スキーマ名>;
SET OUTPUT CAPTURE SCRIPT "capture_ccd_ctltable.sql";
SET OUTPUT TARGET SCRIPT "apply_ccd_ctltable.sql";

#SET RUN SCRIPT LATER;
SET RUN SCRIPT NOW STOP ON SQL ERROR ON;

CREATE CONTROL TABLES FOR APPLY SERVER USING MONITOR LIMIT 93600 MONITOR INTERVAL 60000 IN UW TBSPACE <制御表用の表スペース>;
EOF

# アプライ制御表作成
asnclp -f ./create_apply_ctltable.txt

# ターゲットDBに接続してQアプライとQキャプチャーの制御表が作成されたことを確認
db2 connect to TARGETDB user qrep_ccd using xxxxxx
db2 list tables for schema qrep_ccd
db2 list tables for schema asn

制御表の変更

設計したパラメータを制御表に入れ込んでいく。 以下の4つのパラメータに対して入れ込むパラメータは基本的にもともとのQレプリケーション環境と一緒。

  • Qアプライ
    • IBMQREP_APPLYPARMS表
    • IBMQREP_RECVQUEUES表
  • Qキャプチャー
    • IBMQREP_CAPPARMS表
    • IBMQREP_SENDQUEUES表

変更方法は通常のUPDATE文でOK。 db2 "update <制御表スキーマ>.<制御表> set <設定項目>=<設定値> 追加でキャプチャー制御表の「IBMQREP_IGNTRAN表」に以下のレコードをINSERTする。 AITHID列にQrep用ユーザー名を指定することで、Qレプリケーションによる更新はCCD表に書き出されないようになる(無視される)。 これを設定しないと対象表に対する全トランザクションがCCD表に書き出されるので非常にサイズが膨れ上がりもともと探したかった異常な更新も見つけにくくなってしまう。

AUTHID AUTHTOKEN PLANNAME IGNTRANTRC
qrep_target null null N

5. テスト用サブスクリプションの作成と起動

テスト用サブスクリプション作成

CCD環境での「ソース表」はターゲットDBにある監視対象の「ターゲット表」。 そしてCCD環境での「ターゲット表」はターゲットDBに作成する新しいCCD表。 ホスト→DBサーバーのQrepはソース表、ターゲット表をそれぞれ先に作成しておいたが、 今回はCCD表という特殊な表のためターゲット表はasnclpコマンドが生成するSQL文を利用して作成する。 以下は通常のQsubと異なるオプションなので要注意。

  • SET RUN SCRIPT LATER
    今回はターゲット表をQsubと一緒に作成するため、スクリプト実行時にそのまま制御表に書き込ませてしまうとターゲット表の表スペースを指定できない。そのため「SET RUN SCRIPT LATER」とすることでasnclpコマンド実行時に出力されるSQLを手で修正してから手動実行する。
  • EXIST なし
    ターゲット表(CCD表)は事前に作成せずQsub作成時に作成するのでターゲット表が既に存在することを示す「EXIST」オプションは外しておく。
  • HAS LOAD PHASE N
    Qsub起動時にフルリフレッシュを行わない。 非完全のCCD表を構成する場合このオプションは必須。
  • TYPE CCD CONDENSED OFF COMPLETE OFF WITH OUW COLS ALL
    非圧縮(“CONDENSED OFF”)かつ非完全(“COMPLTE OFF”)のCCD表を構成する。 非圧縮は1つのレコードに対して最新状態だけでなくこれまでの全ての変更履歴をレコードとして保持する。 非完全はソース表と同じ行数のレコードを持つのではなく、更新された行のデータのみ持つという意味。
  • CAPTURE LOAD W (デフォルト、指定なし) ソース表へのロード処理を検知したら警告を出す。(フルリフレッシュは行わない)

# Qサブスクリプション作成用スクリプトの準備
cd /home/db2qrep/QREP/define
cat > ./create_qsub_test_ccd.txt << EOF
ASNCLP SESSION SET TO Q REPLICATION;
SET PWDFILE "<パスワードファイル(フルパス)>";

SET SERVER CAPTURE TO DB <ソースDB名>;
SET SERVER TARGET TO DB <ターゲットDB名>;

SET CAPTURE SCHEMA SOURCE <制御表スキーマ>;
SET APPLY SCHEMA <制御表スキーマ>;

SET RUN SCRIPT LATER;
#SET RUN SCRIPT NOW STOP ON SQL ERROR ON;

SET OUTPUT CAPTURE SCRIPT "cap_creqsub_testtable_ccd.sql";
SET OUTPUT TARGET SCRIPT "app_creqsub_testtable_ccd.sql";

CREATE QSUB USING REPLQMAP "<Qマップ名>" (SUBNAME "<Qsub名>" "<ソース表スキーマ名>"."<ソース表名>" OPTIONS SPILL_MODELQ "<スピルキュー名>"
HAS LOAD PHASE N CONFLICT ACTION F ERROR ACTION S REPLICATE ADD COLUMN Y START AUTOMATICALLY NO TARGET NAME "<ターゲット表スキーマ名>"."<ターゲット表名>"
TYPE CCD CONDENSED OFF COMPLETE OFF WITH OUW COLS ALL);
EOF

# 作成したスクリプトを実行(制御表への書き込みはされない)
asnclp -f ./create_qsub_test_ccd.txt

# 生成されたSQLファイルを直接編集してCCD表の表スペースを指定のものに書き換える
vi ./app_creqsub_testtable_ccd.sql

# ターゲットDBへ接続
db2 connect to TARGETDB user qrep_ccd using xxxxxx

# Qsub作成用SQLの実行
db2 -tvf ./app_creqsub_testtable_ccd.sql
db2 -tvf ./cap_creqsub_testtable_ccd.sql

# Qsub確認
db2 "select SUBNAME,STATE from qrep_ccd.ibmqrep_targets"
db2 "select SUBNAME,STATE from qrep_ccd.ibmqrep_subs"

# CCD表の確認
db2 describe table <ターゲット表スキーマ>.<ターゲット表名>
db2 "select * from <ターゲット表スキーマ>.<ターゲット表名>

# Qキャプチャー起動スクリプトの作成
cat /xxx/xxx/ccd_capturestart.sh << EOF
export DB2CODEPAGE=943
export TMPDIR=/home/db2qrep/qrep
nohup asnqcap capture_server=<ターゲットDB名> capture_schema=<制御表スキーマ名> capture_path=/home/db2qrep/qrep startmode="warmsi" &
EOF
chown u+x /xxx/xxx/ccd_capturestart.sh

# Qキャプチャー停止スクリプトの作成
cat /xxx/xxx/ccd_capturestop.sh << EOF
export TMPDIR=/home/db2qrep/qrep
asnqacmd capture_server=<ターゲットDB名> capture_schema=<制御表スキーマ名> stop
EOF
chown u+x /xxx/xxx/ccd_capturestop.sh

# Qアプライ起動スクリプトの作成
cat /xxx/xxx/ccd_applystart.sh << EOF
export DB2CODEPAGE=943
export TMPDIR=/home/db2qrep/qrep
nohup asnqcap apply_server=<ターゲットDB名> apply_schema=<制御表スキーマ名> apply_path=/home/db2qrep/qrep &
EOF
chown u+x /xxx/xxx/ccd_applystart.sh

# Qアプライ停止スクリプトの作成
cat /xxx/xxx/ccd_applystop.sh << EOF
export TMPDIR=/home/db2qrep/qrep
asnqacmd apply_server=<ターゲットDB名> apply_schema=<制御表スキーマ名> stop
EOF
chown u+x /xxx/xxx/ccd_applystop.sh

# Qキャプチャーの起動
/xxx/xxx/ccd_applystart.sh

# Qアプライの起動
/xxx/xxx/ccd_applystart.sh

# プロセスを確認
ps -ef | grep asnqapp | grep -v grep
